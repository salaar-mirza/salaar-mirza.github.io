Project Overview

Neo-Serpent is a modern take on the classic Snake arcade game, featuring a neon-soaked cyberpunk aesthetic, power-ups, and a competitive 1v1 local multiplayer mode.

Beyond the gameplay, this project serves as a case study in Vertical Slice Architecture. The goal was to move away from "spaghetti code" prototypes and build a scalable, testable codebase where features (like Snake, Food, and GameLoop) are self-contained and highly cohesive.

Architecture: Vertical Slices

Traditional Unity projects often layer applications horizontally (e.g., Scripts/UI, Scripts/Managers). In Neo-Serpent, I organized the codebase by Feature.

This means everything related to a specific feature—Logic, View, Data, and Spawners—lives together in one folder. This approach significantly reduces coupling and makes adding new features (like a new Power-Up) much faster and safer.

The "God Class" Refactor

A common pitfall in game development is the massive GameManager.cs that does everything. I refactored this responsibility into three distinct classes to adhere to the Single Responsibility Principle:

  • GameBootstrap (Builder): Responsible only for wiring dependencies and initialization.
  • GameLoopManager (Engine): Handles the execution loop and time accumulation.
  • CollisionSystem (Rules): Pure logic class determining Win/Loss conditions.

Decoupling Logic from View

The Domain Logic (Snake.cs) knows nothing about Unity. It uses pure C# events to communicate state changes. This allows the core game logic to be unit-tested without the overhead of the Unity Editor. 

// Snake.cs (Domain Logic) - No UnityEngine dependencies
public class Snake
{
    public event Action OnMove;
    public event Action<int> OnGrow;

    public void Move()
    {
        // ... Calculate new position ...
        OnMove?.Invoke();
    }
}

// SnakePresenter.cs (View) - Handles Unity specifics
public class SnakePresenter : MonoBehaviour
{
    public void Initialize(Snake snake)
    {
        // Observer Pattern: React to logic changes
        snake.OnMove += UpdateVisuals;
    }

    private void UpdateVisuals() { /* Update Transform/Sprites */ }
}

Input Abstraction & Strategy Pattern

To support both Single Player and Local Multiplayer (where P1 uses WASD and P2 uses Arrow Keys), I implemented an IInputProvider interface.

This Strategy Pattern allows the game to hot-swap control schemes or even plug in an AI controller without touching the core Snake logic.

Trade-offs: OOP vs. ECS

For this project, I deliberately chose Object-Oriented Programming (OOP) over Entity Component System (ECS).

Reasoning: ECS shines with massive scale (10,000+ objects). Neo-Serpent has ~2 Snakes and ~5 Food items, so the performance overhead of OOP is negligible. Prioritizing clean Domain-Driven Design and maintainability was more valuable for this portfolio piece than raw optimization.
