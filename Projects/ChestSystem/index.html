Project Overview

The Chest System is a modular feature designed to handle the acquisition, unlocking, and opening of loot boxes in a mobile-style game environment. Unlike a simple prototype, this project was built with scalability and maintainability in mind.

The core challenge was to decouple the Game Logic (Currency, Timers) from the UI (Popups, Animations) while ensuring a smooth user experience. To achieve this, I implemented a Service Locator architecture combined with a centralized Event Bus.

Architecture: MVC & Services

I moved away from monolithic "Manager" classes and adopted a Service-based approach. The GameService acts as a Singleton entry point, initializing specific services like EventService, CurrencyService, and UIService.

For the chests themselves, I utilized the Model-View-Controller (MVC) pattern:

  • Model: ChestScriptableObject (Data) & ChestModel (Runtime State).
  • View: ChestSlotView (UI references and animations).
  • Controller: ChestSlotController (Handles input and updates the model).

Decoupling with Event Bus

To prevent "Spaghetti Code," components do not reference each other directly. Instead, they communicate via a type-safe Event Service. This allows the UI to react to logic changes without the Logic layer knowing the UI exists. 

// EventService.cs - The Central Hub
public class EventService
{
    // UI Events
    public GameEventController<ChestSlotView> OnChestSelected { get; private set; }
    
    // Logic Events
    public GameEventController<int> OnCurrencyUpdated { get; private set; }
    
    // Input Events (Commands)
    public GameEventController OnUndoCommand { get; private set; }

    public EventService()
    {
        OnChestSelected = new GameEventController<ChestSlotView>();
        OnCurrencyUpdated = new GameEventController<int>();
        OnUndoCommand = new GameEventController();
    }
}

Command Pattern & Undo System

One of the complex requirements was the ability to "Undo" spending gems to unlock a chest. I implemented the Command Pattern to encapsulate actions as objects.

When a user unlocks a chest with gems, an InstantUnlockCommand is created and executed. This command stores the previous state, allowing the system to revert the transaction (refund gems, re-lock chest) if the user clicks Undo. 

// InstantUnlockCommand.cs
public class InstantUnlockCommand : ICommand
{
    private ChestSlotController _chestSlot;
    private int _cost;

    public void Execute()
    {
        CurrencyService.Instance.DeductGems(_cost);
        _chestSlot.Unlock();
    }

    public void Undo()
    {
        CurrencyService.Instance.AddGems(_cost); // Refund
        _chestSlot.Lock(); // Revert State
    }
}

Data-Driven Design

The system is designed to be designer-friendly. All chest data (names, icons, coin ranges, unlock times) are stored in ScriptableObjects. This allows game designers to create new chest types or balance the economy without touching a single line of code.

Optimization: Object Pooling

To handle visual effects (like rain or coin explosions) efficiently, I implemented a generic Object Pool. This prevents memory fragmentation and Garbage Collection spikes by reusing objects instead of constantly Instantiating and Destroying them.
